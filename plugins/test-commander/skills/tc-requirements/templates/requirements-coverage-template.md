# Requirements Coverage

The structure `/tc:requirements-coverage` generates. The helper writes a file matching this shape to `<workspace>/requirements/requirements-coverage.md` on every run (overwriting any prior content; the file is generated, not hand-edited). A parallel traceability map at `<workspace>/traceability/requirements-map.md` carries the same coverage rows in a leaner format suited to cross-skill consumption.

## Executive summary

- Requirements: **N**
- Covered (any downstream artifact): **M**
- Not yet covered: **K**
- Orphan downstream artifacts: **J**

## Coverage matrix

A row per requirement in inventory order. Each downstream-artifact column lists the relative paths (under `.test-commander/`) that link back to the requirement, or `_(none)_`.

| REQ-ID | Test ideas | BDD features | Automation |
| --- | --- | --- | --- |
| REQ-001 | `test-ideas/REQ-001.md` | _(none)_ | _(none)_ |
| REQ-002 | _(none)_ | _(none)_ | _(none)_ |
| ... | ... | ... | ... |

## Not yet covered

Listed when at least one requirement has no downstream artifact.

> _N requirement(s) have no downstream artifact yet. Phases 4-6 populate the downstream surfaces; Step 2.6 lands seed test ideas._

- REQ-NNN
- REQ-NNN

## Orphan downstream artifacts

Listed when at least one downstream artifact references a REQ-ID that is not in the inventory. Each row names the source artifact, its kind (`test-ideas` / `bdd` / `automation`), and the unresolved REQ-ID target.

| Source | Kind | Target |
| --- | --- | --- |
| `test-ideas/REQ-999.md` | `test-ideas` | REQ-999 |

> _Resolve orphans before re-running coverage: either restore the referenced REQ, rename the artifact, or remove the artifact._

## Traceability

Inventory IDs in document order, comma-separated, as a quick sanity check that the coverage file is using the same ID space the inventory produced.

`REQ-001, REQ-002, ..., REQ-NNN`

## Parallel traceability map (`traceability/requirements-map.md`)

The helper also writes a leaner file containing only the coverage matrix and a one-paragraph header. Downstream skills (Phases 3, 4, 5, 6, 7) consume this map without needing to parse the executive summary or orphan list.

```
# Requirements Map

Cross-references each parsed requirement to the downstream artifacts that
trace back to it. Generated by `/tc:requirements-coverage` (Step 2.5);
overwritten on every run.

| REQ-ID | Test ideas | BDD features | Automation |
| --- | --- | --- | --- |
| REQ-001 | `test-ideas/REQ-001.md` | _(none)_ | _(none)_ |
| ...    | ...                     | ...      | ...      |
```

## Notes for human reviewers

This template is the contract for the **mechanical** output. The helper does no judgment — it lists every downstream link it can find. Claude's narrative layer:

- Flags requirements that are formally "covered" but only by speculative test ideas (Step 2.6 seeds), not real BDD or automation.
- Recommends which uncovered requirements should be addressed first (risk-aware prioritization).
- Surfaces orphans for resolution.

The mechanical output is deterministic and idempotent; the narrative layer is rendered per session.
