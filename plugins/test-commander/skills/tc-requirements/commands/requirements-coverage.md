# `/tc:requirements-coverage`

Cross-reference the requirement IDs in `.test-commander/requirements/requirements-inventory.md` with the downstream artifacts that should trace back to them (test ideas, BDD scenarios, automation map) and write a coverage matrix plus a parallel traceability map. This page is the authoritative spec.

## Inputs

- **Project root** (positional, optional, default = current working directory). Must contain a `.test-commander/` workspace (run `/tc:init` first if absent).
- **Inventory artifact.** `<project-root>/.test-commander/requirements/requirements-inventory.md` produced by `/tc:review-requirements` (Step 2.2). The unmodified workspace template stub does NOT count — the helper requires the inventory to bear the Step 2.2 generator's markers.
- **Downstream artifacts** (all optional, all parsed if present):
  - `<workspace>/test-ideas/*.md` — files named `REQ-NNN.md` link by convention, and any `REQ-\d+` reference in a body links the file to that requirement.
  - `<workspace>/bdd/features/*.feature` — any `REQ-\d+` reference (typically in a comment or tag) links the feature to that requirement.
  - `<workspace>/traceability/automation-map.md` — any `REQ-\d+` reference links the requirement to the automation surface (Phase 6 owns the canonical map).

## Outputs

Two files:

| Path | Behavior |
| --- | --- |
| `requirements/requirements-coverage.md` | **Overwritten** on every run. Full coverage report — executive summary, coverage matrix, not-yet-covered list, orphan list, traceability footer. |
| `traceability/requirements-map.md` | **Overwritten** on every run. Leaner ID-to-downstream-links map for downstream-skill consumption. |

The coverage file follows the structure documented in [`templates/requirements-coverage-template.md`](../templates/requirements-coverage-template.md).

## Preconditions

- `.test-commander/` exists at the project root (`/tc:init` has been run).
- `.test-commander/requirements/requirements-inventory.md` has been generated by `/tc:review-requirements`. If the file is the unmodified template stub, the helper refuses with `InventoryMissingError`.

## Behavior

1. **Validate workspace.** If `<project>/.test-commander/` is missing, raise `UninitializedWorkspaceError`, exit 2.
2. **Validate inventory.** If the inventory file is absent or shows only the template stub content, raise `InventoryMissingError`, exit 2.
3. **Parse inventory.** Extract REQ-IDs from the inventory's Markdown table in document order. This is the *known* ID space.
4. **Scan downstream artifacts.**
   - `test-ideas/*.md`: filename matching `REQ-NNN.md` links by convention; `REQ-\d+` references in bodies link the file to those IDs.
   - `bdd/features/*.feature`: `REQ-\d+` references in any line link the feature.
   - `traceability/automation-map.md`: `REQ-\d+` references link the requirement.
5. **Assemble coverage.** For each REQ-ID in the inventory, build a `CoverageLink` of the linked downstream paths per kind. A REQ-ID is *covered* iff it has at least one downstream link of any kind.
6. **Detect orphans.** Every downstream reference whose target REQ-ID is not in the inventory becomes an `Orphan` entry. The orphan list is sorted by `(target, kind, source)`.
7. **Write artifacts.** Overwrite `requirements/requirements-coverage.md` and `traceability/requirements-map.md`. Both are byte-deterministic for the same input.
8. **Print CLI summary** to stdout and exit 0.

## Safety

- Read-only outside `.test-commander/requirements/` and `.test-commander/traceability/`. Does not touch source code, BDD specs (only reads them), test code, or any external system.
- Network access: none.
- Idempotent: re-running against unchanged input produces byte-identical coverage and traceability-map files.
- Workspace-bounded: every path resolved relative to the supplied project root.

## Implementation

`plugins/test-commander/scripts/requirements_coverage.py` (bundled with the plugin per D18).

CLI:

```sh
python3 <plugin-root>/scripts/requirements_coverage.py [project_root]
```

`project_root` defaults to the current working directory.

When invoked via the `/tc:requirements-coverage` slash command, Claude resolves `<plugin-root>` from the `tc-requirements` SKILL.md location, runs the helper via `Bash`, and reports the helper's CLI output plus a narrative summary that highlights uncovered requirements and orphans.

## Definition of done

- Helper passes `tests/test_requirements_coverage.py` (8 cases, including orphan detection and traceability-map ID parity).
- Template doc describes the output structure for both the coverage file and the parallel traceability map.
- SKILL.md describes the shipped behavior with no deferral wording for this command.
- Coverage shape is forward-compatible with Phase 4 (test ideas), Phase 5 (BDD features), and Phase 6 (automation map) artifacts.
