# Test data strategy

How Test Commander manages test data so it never lives inside test code
(Decision D6). `/tc:generate-test-data` populates the `.test-commander/test-data/`
tree from the BDD scenarios; the generated fixtures (`/tc:automate`) reach that
data through a file, never inlining literals.

## Why data lives outside the tests (D6)

Inlined credentials, payloads, and records make tests brittle and untraceable: a
data change means editing code, and the same value is duplicated across specs.
Test Commander keeps data declarative and addressable under `test-data/`, loaded
by a fixture. A spec that inlines data is an `inline-test-data` finding in the
automation review.

## The three surfaces (Q11)

| Directory | Format | Role |
| --- | --- | --- |
| `test-data/seed/<area>.json` | JSON fixture | The file the generated fixture loads. One record per `@cs:` candidate scenario. |
| `test-data/scenarios/<area>.md` | Markdown spec | The declarative data requirement per scenario - what records each scenario needs, in prose. |
| `test-data/factories/` | Python factories | Authored by hand only where declarative data is insufficient (computed values, large volumes, sequences). Never generated or overwritten. |

Declarative formats (JSON + Markdown) cover the universal case; a Python factory
is the exception, not the default.

## Overwrite vs preserve

- **Generated files are overwritten.** A file carrying the generated marker
  (`_generated_by: /tc:generate-test-data` in JSON, "Generated by
  /tc:generate-test-data" in Markdown) is regenerated each run, byte-stable
  against unchanged scenarios.
- **User-authored files are preserved.** A file with no generated marker is
  hand-authored and is skipped, never clobbered. This is how a project fills in
  real field values: edit the generated seed (removing or keeping the marker as
  you choose) or drop in a marker-less file, and regeneration leaves it alone.

## The universal shipped shape (D19)

The generated seed JSON is generic: records keyed by candidate id, each with
`id`, `requirement`, and `label`. There are no product-specific fields - a
consuming project fills in the real values by hand (preserved) or via a factory.
The shipped defaults name no domain.

## The Claude judgment layer

The generated seed is a scaffold with placeholder records. Claude reads the
scenarios and `product-knowledge/` to flesh the records out with realistic,
scenario-appropriate values - valid and invalid credentials for a sign-in
scenario, boundary values for a limits scenario - shaping the data the fixture
serves. The generator guarantees the file exists and is addressable; Claude
makes the data meaningful.
