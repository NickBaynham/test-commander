# Automation review

How Test Commander reviews generated Playwright specs for quality and
maintainability. `/tc:review-automation` scores every spec under `tests/e2e/`
against six universal categories (Decision D19), writes a per-spec verdict, and
routes failures to the open-questions log. The same engine auto-runs at the end
of `/tc:automate`.

## The six categories

| Category | Signal | Why it matters |
| --- | --- | --- |
| `inline-test-data` | a literal passed to `.fill`/`.type`/`.selectOption` | Data belongs in the `.test-commander/test-data/` tree, reached via the fixture (D6); inlined data is unmaintainable and untraceable. |
| `hardcoded-wait` | `waitForTimeout` / `sleep` | Fixed sleeps are the primary source of flake; rely on Playwright auto-waiting and web-first assertions. |
| `missing-provenance` | a `test()` with no `// @req:`/`@cs:` comment | Without provenance the test cannot be traced to a requirement or candidate, breaking the traceability chain. |
| `weak-locator` | a CSS/XPath selector (`.locator('#'`/`'.'`, `xpath=`, `page.$`) | Structural selectors break on markup churn; prefer role/label/test-id (see the locator strategy). |
| `untraceable-spec` | the spec file is not linked in `automation-map.md` | An orphan spec is not part of the generated, traceable suite; regenerate it via `/tc:automate`. |
| `assertion-free` | a `test()` with no `expect(` call | A test that asserts nothing cannot fail meaningfully; it gives false confidence. |

One finding per category per spec file. A clean spec generated by `/tc:automate`
violates none of them.

## Verdict and routing

- A per-spec verdict is written to
  `<workspace>/automation-plan/review-summary.md` (`pass`, or
  `N finding(s) - categories: ...`).
- Each finding is routed to `<workspace>/requirements/open-questions.md` as a
  `[automation-review]` gap signal, deduplicated by per-spec source-id
  (`tc-automate/automation-review-<area>`) + question text - the Phase-2
  open-questions contract. Re-running is idempotent.

## Shared engine, auto-run, and suppression

`review_automation()` is a single function. The standalone
`/tc:review-automation` command and the `/tc:automate` generate-time auto-run
both call it, so a spec is judged identically however the review is triggered.
`/tc:automate` runs it automatically after writing the automation map; pass
`--no-review` to suppress the auto-run (for example, when generating in bulk and
reviewing once at the end).

## The Claude judgment layer

The rubric is mechanical and conservative - it flags structural smells, not
semantic correctness. After the review, Claude reads the summary and the flagged
specs to decide what to fix: it rewrites inline data into fixture-backed data,
replaces brittle locators, adds the missing provenance, and strengthens vacuous
assertions into real expected outcomes. The rubric guarantees the smells are
surfaced; Claude's judgment resolves them.
