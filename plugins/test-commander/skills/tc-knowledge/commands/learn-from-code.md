# `/tc:learn-from-code`

The Phase 3 source-code ingestion command. Walks Python files under the configured source root (default `<workspace>/documents/uploaded/code/`) using stdlib `ast`, extracts modules / classes / functions / docstrings / decorators with file:line provenance, routes gap signals (`undocumented-function`, `language-unsupported-in-v1`, `unimplemented-endpoint`) to `<workspace>/requirements/open-questions.md`, contributes a `## From code` section to `entities.md` (classes as domain entities), and regenerates `system-model.md` via the shared `synthesize_system_model.py` helper.

## Inputs

- Python `*.py` files under `<workspace>/<source-root>/`, default source root `documents/uploaded/code/`. Configurable via `tc-knowledge.code.source-root`.
- Non-Python files under the same root with extensions `.ts`, `.tsx`, `.js`, `.jsx`, `.go`, `.java`, `.rb` are detected and flagged as `language-unsupported-in-v1` gaps. They are not parsed in v1.
- `<workspace>/config.yaml` (optional) - the `tc-knowledge.code:` block extends extraction. Recognized keys:
  - `source-root: src` - relative to workspace root. Default `documents/uploaded/code`.
  - `enabled-languages: [python]` - v1 parses python only; the key is documented for future-phase extension.
  - `ignored-paths: [migrations, __pycache__, .venv]` - substring matching against the relative path; default includes `__pycache__`, `.git`, `.venv`, `node_modules`.
  - `endpoint-decorator-patterns: [...]` - reserved for the v2 spec-vs-decorator cross-check; v1 ignores.
- `<workspace>/product-knowledge/spec-derived-model.md` (optional) - if generated by `/tc:learn-from-specs`, the helper cross-checks spec endpoints against parsed function names and emits `unimplemented-endpoint` gaps for any endpoint whose `operationId` is not a function name in the code.

## Outputs

| Path | Mode | Owner |
| --- | --- | --- |
| `<workspace>/product-knowledge/code-derived-model.md` | overwrite | this command |
| `<workspace>/product-knowledge/entities.md` (`## From code`) | section-overwrite | this command |
| `<workspace>/product-knowledge/business-rules.md` (`## From code`) | section-overwrite (empty in v1) | this command |
| `<workspace>/product-knowledge/system-model.md` | overwrite (synthesizer) | shared |
| `<workspace>/requirements/open-questions.md` | append, dedup by `(source-id, question)` | this command |

`user-journeys.md` and `assumptions.md` are NOT touched; code declares no journeys and code facts are confirmed not inferred. The `business-rules.md` `## From code` section is empty in v1 because the seeded fixture has no module-level constants carrying explicit rule docstrings; the section therefore does not appear in the rendered file.

## Preconditions

- `<workspace>/.test-commander/` exists. The helper exits non-zero with a clear error if the workspace is uninitialized.

No upstream Phase 3 helper is strictly required. The `unimplemented-endpoint` cross-check fires only when `spec-derived-model.md` is also generated, so running `/tc:learn-from-specs` first unlocks that gap detector.

## Behavior

1. Resolve the workspace under `<project-root>/.test-commander/`.
2. Load `tc-knowledge.code:` extensions from `<workspace>/config.yaml`; fall back to documented defaults.
3. Resolve the source root: `<workspace>/<source-root>` (e.g., `documents/uploaded/code`).
4. Walk every `*.py` file under the source root, skipping any whose relative path contains a token from `ignored-paths`. Parse each with `ast.parse`; capture module + classes (with base classes and `__init__`-derived attributes) + functions (with decorators rendered back to source-like strings).
5. Walk the same root for non-Python files in the unsupported-extension set; emit `language-unsupported-in-v1` gaps.
6. For every public function (name not starting with `_`, and `Class.method` form where `method` does not start with `_`) without a docstring, emit `undocumented-function`.
7. If `spec-derived-model.md` is generated (carries the `Auto-generated by /tc:learn-from-specs` marker and is not the empty-run stub), parse its endpoints table for `(method, path, operationId)` triples; for each operationId not present in the function-name set, emit `unimplemented-endpoint`.
8. Render `code-derived-model.md` (overwrite) with source modules + classes + functions + unsupported-files + gap-signals sections.
9. Render the `## From code` section body for `entities.md` (classes as domain entities) and `business-rules.md` (empty in v1 -> section omitted by the renderer); call `update_cross_cutting()` which preserves every other source's section.
10. Append open questions for each gap signal, deduplicated by `(source-id, question-text)` (the question text includes the gap kind as a prefix).
11. Call `synthesize_system_model.synthesize(project_root)` to regenerate `system-model.md`.
12. Exit 0.

## Safety

- The helper reads only from the configured source root. No network calls; no shell-out; no writes outside `product-knowledge/` and `requirements/open-questions.md`.
- `ast.parse` is the standard CPython parser; it does not execute code, only parses syntax.
- A `SyntaxError` in any source file causes that single file to be skipped silently; the helper continues to the next file. A corrupt Python file is not allowed to crash the run.
- Re-running against unchanged input is byte-deterministic across every output. The `open-questions.md` dedup prevents append drift.
- Per the Phase-3 design decisions, this helper does not write to `<workspace>/traceability/`.
- Per D19, the language detector and the keyword sets are universal cores; project-specific tuning happens through `tc-knowledge.code:` config extensions.

## Implementation

- Helper: `plugins/test-commander/scripts/extract_knowledge_from_code.py` (~700 lines).
- Tests: `tests/test_learn_from_code.py` (23 cases - uninitialized refused, no-code stub, AST extraction (classes + functions + docstrings + provenance), all three gap signals route to open-questions, `unimplemented-endpoint` skipped without spec model, cross-cutting scope discipline (entities touched; user-journeys + assumptions NOT touched), prior-section preservation, full idempotency, `source-root` and `ignored-paths` extensions, synthesizer reflects code).

## Definition of Done

- Helper passes all 23 test cases.
- AST walk covers modules / classes / functions / docstrings / decorators with provenance.
- Cross-check against `spec-derived-model.md` works when present and is skipped when absent.
- Non-Python file detection is explicit (not silently ignored).
- Methodology covers all five positive dimensions plus all three gap signals with worked examples and Claude-judgment-layer paragraphs.
- Template authored.
- Per-command page complete (this file).
- `tc-knowledge/SKILL.md` describes `/tc:learn-from-code`'s shipped behavior; no deferral wording for this command.
- `make verify` chain green.
