# `/tc:learn-from-api`

The Phase 3 runtime-API ingestion command. Reads a recorded-response playback file (default `<workspace>/documents/uploaded/recorded-api/responses.json`), extracts live endpoints with status families, captures response-body shapes, infers auth-required endpoints from Authorization headers and 401/403 responses, routes gap signals (`unspecified-endpoint`, `mismatched-status`) to `<workspace>/requirements/open-questions.md`, contributes a `## From api` section to `entities.md` (resources confirmed reachable) and `business-rules.md` (auth-required rules), and regenerates `system-model.md` via the shared synthesizer. Live mode is opt-in and refused under pytest.

## Inputs

- `<workspace>/documents/uploaded/recorded-api/responses.json` (default; override via `tc-knowledge.api.recorded-path`) - a JSON list of `{method, path, status, headers, body}` objects.
- `<workspace>/product-knowledge/spec-derived-model.md` (optional) - if generated by `/tc:learn-from-specs`, the helper cross-checks every recorded `(method, path, status)` triple against the spec endpoints and their declared response codes.
- `<workspace>/config.yaml` (optional) - the `tc-knowledge.api:` block configures the helper. Recognized keys:
  - `mode: recorded` (default) or `mode: live` (refused under pytest).
  - `recorded-path: documents/uploaded/recorded-api/responses.json` (workspace-relative).
  - `base-url: http://localhost:8000` (live mode only).
  - `auth-header: "Authorization: Bearer ${TC_API_TOKEN}"` (live mode only).

## Outputs

| Path | Mode | Owner |
| --- | --- | --- |
| `<workspace>/product-knowledge/api-model.md` | overwrite | this command |
| `<workspace>/product-knowledge/entities.md` (`## From api`) | section-overwrite | this command |
| `<workspace>/product-knowledge/business-rules.md` (`## From api`) | section-overwrite | this command |
| `<workspace>/product-knowledge/system-model.md` | overwrite (synthesizer) | shared |
| `<workspace>/requirements/open-questions.md` | append, dedup by `(source-id, question)` | this command |

`user-journeys.md` and `assumptions.md` are NOT touched - API recordings encode no journeys and are confirmed facts, not inferences.

## Preconditions

- `<workspace>/.test-commander/` exists.
- For the cross-check gaps (`unspecified-endpoint`, `mismatched-status`) to fire, `spec-derived-model.md` must be generated by `/tc:learn-from-specs`. Without it, the cross-check is skipped entirely.

## Behavior

1. Resolve the workspace and load `tc-knowledge.api:` extensions.
2. If `mode: live`, refuse with exit 2:
   - If `PYTEST_CURRENT_TEST` is in the environment, the error names pytest as the reason (no network calls leak from the test suite).
   - Otherwise, the error notes that live mode is not implemented in v1.
3. In recorded mode, resolve the recorded path (default `documents/uploaded/recorded-api/responses.json`); if absent or unparseable, write a `_No recorded API responses found_` stub for `api-model.md` and skip to step 8.
4. For each playback entry, capture `Recording(method, path, status, has_auth_header, response_keys, source_file, index)`. JSON bodies surface their top-level keys; arrays of objects surface the first element's keys. Detect `auth-required` when the request carries an `Authorization` header or the response is 401 / 403 without one.
5. If `spec-derived-model.md` is generated, call `extract_knowledge_from_specs.aggregate()` (Step 3.3's parser) and build a `(method, path) -> declared_statuses` map. For each recording, emit `unspecified-endpoint` if the key is absent, or `mismatched-status` if the recorded status is outside the declared set (and the declared set is non-empty).
6. Render `api-model.md` (overwrite) with source-recordings table + live-endpoints table + response-shapes table + auth-required list + gap-signals section.
7. Render the `## From api` section body for `entities.md` (resources confirmed reachable, grouped by first non-templated path segment) and `business-rules.md` (one rule per auth-required endpoint); call `update_cross_cutting()` which preserves every other source's section.
8. Append open questions for each gap signal, deduplicated by `(source-id, question-text)` (the question text includes the gap kind as a prefix).
9. Call `synthesize_system_model.synthesize(project_root)` to regenerate `system-model.md`.
10. Exit 0.

## Safety

- Live mode is refused under pytest before any HTTP request is constructed. The refusal is asserted by `test_live_mode_refused_under_pytest`.
- Recorded mode reads only the configured playback file. No shell-out; no network; no writes outside `product-knowledge/` and `requirements/open-questions.md`.
- `json.loads` is used for all body parsing; arbitrary code execution is impossible.
- Re-running against unchanged input is byte-deterministic across every output. The `open-questions.md` dedup prevents append drift.
- Per the Phase-3 design decisions, this helper does not write to `<workspace>/traceability/`.
- Per D19, the dimension detectors carry no domain vocabulary.

## Implementation

- Helper: `plugins/test-commander/scripts/extract_knowledge_from_api.py` (~600 lines).
- Reuses `extract_knowledge_from_specs.aggregate()` for the cross-check (via the `Endpoint.statuses` field added in Step 3.5).
- Tests: `tests/test_learn_from_api.py` (23 cases - uninitialized refused, no-recorded stub, recorded extraction (7 entries + provenance + response shapes + auth-required), both gap signals route to open-questions, gap signals skipped without spec model, cross-cutting scope discipline (entities + business-rules touched; user-journeys + assumptions NOT touched), prior-section preservation, full idempotency, recorded-path extension, live mode refused under pytest, synthesizer reflects api).

## Definition of Done

- Helper passes all 23 test cases.
- Recorded mode is the test contract; live mode is documented but never reached by tests.
- Cross-check against `spec-derived-model.md` works when present and is skipped when absent.
- Methodology covers all 3 positive dimensions plus both gap signals with worked examples and Claude-judgment-layer paragraphs.
- Template authored.
- Per-command page complete (this file).
- `tc-knowledge/SKILL.md` describes `/tc:learn-from-api`'s shipped behavior; no deferral wording for this command.
- `make verify` chain green.
