# Project knowledge (umbrella methodology)

This document describes how the five `/tc:learn-from-*` commands compose into a single coherent picture of the consuming project's product, contracts, source, runtime behavior, and existing test coverage. It is the cross-source contract the per-command methodology docs (`learning-from-documents.md`, `learning-from-specs.md`, etc.) sit underneath.

## The synthesis model

Test Commander's knowledge layer has three artifact tiers:

1. **Per-source models** under `<workspace>/product-knowledge/`:
   - `documentation-model.md` - populated by `/tc:learn-from-docs`.
   - `spec-derived-model.md` - populated by `/tc:learn-from-specs`.
   - `code-derived-model.md` - populated by `/tc:learn-from-code`.
   - `api-model.md` - populated by `/tc:learn-from-api`.
   - `tests-coverage.md` - populated by `/tc:learn-from-tests`.

   Each command writes its file byte-deterministically (overwrite on every run). The file is the helper's narrative report against its single source.

2. **Cross-cutting artifacts** under the same directory:
   - `entities.md`
   - `user-journeys.md`
   - `business-rules.md`
   - `assumptions.md`

   Each is structured as a header plus one or more `## From <source>` sections. Every `/tc:learn-from-*` command owns its own section (`## From documents`, `## From specs`, etc.) and re-writes only that block. Sections from other commands are preserved across re-runs. The stable section order in the rendered file is `documents -> specs -> code -> api -> tests`.

3. **Synthesized overview** at `system-model.md`. Regenerated by `synthesize_system_model.py` at the end of every `/tc:learn-from-*` run. Reads the current state of every per-source model and every cross-cutting artifact and rewrites `system-model.md` from a deterministic template. The synthesis is honest about what has and has not been ingested: per-source models that still carry the workspace-template stub marker, or any "no <source> found" sentinel a helper emits on the empty path, count as **not yet ingested**.

## Provenance contract

Every fact in any product-knowledge artifact carries a citation. The citation format depends on the source type:

- Documents, specs, code, tests - `<relative-path>:<line>` or `<relative-path>:<start>-<end>`.
- Live or recorded API - `<METHOD> <path> <status>` (the request line plus the observed status code).

Cross-cutting artifact bullets carry the citation inline; the per-source models carry the citation in the rightmost column of every findings table. The synthesizer does not invent provenance; if it cannot trace a name back to a section in a cross-cutting file, it does not list the name.

## Assumptions vs confirmed facts

A confirmed fact has a direct citation in the source text (a glossary definition, a `class` statement, an OpenAPI `paths` entry, a recorded `2xx` response). An assumption is Claude's narrative inference - "the documentation implies that X holds" - without an exact citation. Assumptions are written under the `## From <source>` section of `assumptions.md` and tagged `no direct citation` in the rendered text. The methodology forbids promoting an assumption to an entity, journey, or rule without also writing the citation that converted it.

## Gap-signal routing

Every learn helper detects gap signals against its own partition table:

- `undefined-term`, `contradictory-rule` from documents.
- `unspecified-status`, `schema-without-type` from specs.
- `unimplemented-endpoint`, `undocumented-function`, `language-unsupported-in-v1` from code.
- `unspecified-endpoint`, `mismatched-status` from the API.
- `untested-function`, `unsupported-test-runner` from tests.

Gaps land in two places. The per-source model lists them in its `## Gap signals` section. They are also appended to `<workspace>/requirements/open-questions.md`, deduplicated by `(source-id, question-text)`, using the Phase-2 contract. User-authored questions in that file are preserved.

## Workspace boundaries

Phase 3 only writes to `product-knowledge/` and `requirements/open-questions.md`. It does **not** write to `<workspace>/traceability/`. Cross-source traceability (a requirement linked to an entity linked to an endpoint linked to a test) is Phase 5's responsibility (`/tc:traceability-map`). The Phase 3 helpers supply the inputs Phase 5 will join later; pre-populating the traceability map from Phase 3 skews the `/tc:next` heuristic, per the Phase-2 Step-2.9 lesson.

## Claude's judgment layer

The helpers do the deterministic, universal-vocabulary parts. Claude wraps them with the parts that require product context:

- Synthesizing a coherent narrative across the per-source models when their facts disagree.
- Ranking entity, journey, and rule importance.
- Deciding when an extracted candidate is an entity vs an attribute of an existing entity.
- Explaining *why* a gap signal matters (which feature is at risk, which downstream artifact will be incorrect).
- Translating one source's vocabulary into another (e.g., the spec's `account_id` is the document's `Account`).

Claude reads `system-model.md` first when answering cross-source questions, then descends into the per-source model that owns the relevant facts.
