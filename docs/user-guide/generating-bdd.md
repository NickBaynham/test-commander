# Workflow — Generating BDD and Traceability (Phase 5)

This guide walks you through Test Commander's three Phase 5 commands end to end against a consuming project. The examples use the deliberately-generic seeded fixture from `tests/fixtures/seeded-bdd/` (a Phase-4-enriched `REQ-001.md` test idea and its session summary) plus a small requirements inventory, so every output is reproducible.

## What's available in Phase 5

Phase 5 ships two skills — `tc-bdd` (generation + review) and `tc-traceability` (the cross-cutting map):

| Command | Reads | Writes |
| --- | --- | --- |
| `/tc:generate-bdd` | `test-ideas/REQ-*.md` (enriched seeds) + `sessions/` + `product-knowledge/` | `bdd/features/<area>.feature`, `bdd/summaries/<area>.md`, `bdd/index.md`; auto-runs the review sub-mode (suppressible with `--no-review`) |
| `/tc:review-bdd` | `bdd/features/*.feature` | review verdict in each `bdd/summaries/<area>.md`; appends `[bdd-review]` gap signals to `requirements/open-questions.md` |
| `/tc:traceability-map` | `requirements/requirements-inventory.md` + `test-ideas/` + `bdd/features/*.feature` | `traceability/requirements-map.md` + `traceability/test-map.md` |

Phase 5 **reads broadly** but **writes only to `bdd/`, `traceability/`, and the `[bdd-review]` line in `open-questions.md`**. It does not write `test-ideas/` (Phase 4 owns enrichment) or `product-knowledge/` (Phase 3).

Per Decision D19 ([planning/plan.md](../../planning/plan.md)) the shipped tag classes and review rubric are universal. Domain values (`@area:`/`@risk:`/`@persona:`) and rubric extensions enter through `<workspace>/config.yaml` — see [customizing-for-your-project.md](customizing-for-your-project.md).

## The traceability chain

```
Requirement -> Test Idea -> BDD Scenario -> Automation Candidate
            -> Automated Test -> Test Result -> Quality Report
```

Phase 5 populates the first three links. `/tc:generate-bdd` stamps each scenario with machine-readable `@req:REQ-NNN` and `@cs:CS-NNN-NNN` linkage tags; `/tc:traceability-map` parses those tags to rebuild the maps. Links a later phase owns render `pending`.

## Prerequisites

1. `<workspace>/.test-commander/` exists (`/tc:init` has run — see [workflow.md](workflow.md)).
2. Phase 2 produced a requirements inventory at `<workspace>/requirements/requirements-inventory.md` (via `/tc:review-requirements`). See [reviewing-requirements.md](reviewing-requirements.md).
3. Phase 4 enriched at least one test-idea seed at `<workspace>/test-ideas/REQ-*.md` (`status: enriched`, with a `## Phase 4 enrichment` section). See [exploring-an-app.md](exploring-an-app.md). The seeded `tests/fixtures/seeded-bdd/REQ-001.md` is a worked example of this state.

The natural order is Phase 2 → Phase 3 → Phase 4 → Phase 5: generate BDD only after `/tc:test-ideas` has enriched the seeds.

## Step 1: `/tc:generate-bdd`

Reads each enriched test-idea seed and renders one Gherkin `Scenario` per Phase-4 enrichment candidate (`CS-NNN-NNN`), tagging each for traceability and classification. Writes a per-feature summary and rebuilds the feature index, then auto-runs the review sub-mode.

**Run:**

```sh
python3 <plugin-root>/scripts/generate_bdd.py <project-root> [--req REQ-NNN] [--no-review]
```

**Sample output** (from the seeded fixture):

```
features: 1 (scenarios: 3)
  - .test-commander/bdd/features/account-sign-in-with-email-and-password.feature
```

**What lands** — `bdd/features/account-sign-in-with-email-and-password.feature`:

```gherkin
# Generated by /tc:generate-bdd from test-ideas/REQ-001.md
# Source candidates: SESS-20260115-001
# Deterministic scaffold - refine Given/When/Then phrasing and promote to
# Scenario Outline where data-driven, per methodology/bdd-generation.md.

@area:account-sign-in-with-email-and-password
Feature: Account sign-in with email and password

  @area:account-sign-in-with-email-and-password @req:REQ-001 @cs:CS-001-001 @regression @exploratory @anomaly:auth-mismatch
  Scenario: Expired session routes the holder back to sign-in
    Given an account in the "Account sign-in with email and password" context
    When the negative behavior "Expired session routes the holder back to sign-in" is exercised
    Then the outcome described by "Expired session routes the holder back to sign-in" holds
```

(The feature carries one scenario per candidate; `CS-001-002` and `CS-001-003` follow.) Class tags map from the candidate type — `negative`/`edge` → `@regression`, `happy`/`positive` → `@smoke` — and anomaly-derived candidates also carry `@exploratory @anomaly:<category>`.

The helper emits a deterministic scaffold; Claude refines the Given/When/Then into domain-grounded steps, promotes data-driven scenarios to `Scenario Outline`, and adds `@risk:`/`@persona:` values. Full methodology: [`tc-bdd/methodology/bdd-generation.md`](../../plugins/test-commander/skills/tc-bdd/methodology/bdd-generation.md).

**The per-feature summary** (`bdd/summaries/account-sign-in-with-email-and-password.md`) after the auto-review:

```
- Feature: Account sign-in with email and password
- Requirement: REQ-001
- Scenarios: 3
- Candidates realized: CS-001-001, CS-001-002, CS-001-003
- Sessions: SESS-20260115-001
- Review verdict: pass
```

The `Review verdict: pass` line is written by the auto-run review sub-mode. With `--no-review` it stays `(pending /tc:review-bdd)`.

## Step 2: `/tc:review-bdd`

Runs the six-category universal rubric over every feature: `ambiguous-step`, `missing-tag`, `untraceable`, `ui-coupled-step`, `missing-examples`, `conjunction-overload` (one finding per category per scenario). Writes the verdict into each summary and routes failures to `requirements/open-questions.md` as `[bdd-review]` gap signals. The same implementation is what `/tc:generate-bdd` auto-runs.

**Run:**

```sh
python3 <plugin-root>/scripts/review_bdd.py <project-root>
```

**Sample output** (the generated feature is clean, so it passes):

```
reviewed: 1 feature(s) (0 finding(s))
  - account-sign-in-with-email-and-password: pass
```

When a feature has defects, each finding becomes a line in `requirements/open-questions.md`:

```
- [tc-bdd/bdd-review-<area>] [bdd-review] <category>: scenario '<name>' <message>.
```

Signals are deduplicated by `(source-id, question)`, so re-running never duplicates them. The rubric is the inverse of the generation discipline — see [`tc-bdd/methodology/bdd-quality-review.md`](../../plugins/test-commander/skills/tc-bdd/methodology/bdd-quality-review.md).

## Step 3: `/tc:traceability-map`

Rebuilds the two cross-cutting maps under `traceability/` from the inventory, the enriched test-ideas, and the `@req:`/`@cs:` linkage tags.

**Run:**

```sh
python3 <plugin-root>/scripts/traceability_map.py <project-root>
```

**Sample output:**

```
requirements:     2
scenarios:        3
requirements map: .../traceability/requirements-map.md
test map:         .../traceability/test-map.md
```

**`traceability/requirements-map.md`** — the shared 4-column requirement-to-downstream view (byte-identical to what `/tc:requirements-coverage` writes):

```
| REQ-ID | Test ideas | BDD features | Automation |
| --- | --- | --- | --- |
| REQ-001 | `test-ideas/REQ-001.md` | `bdd/features/account-sign-in-with-email-and-password.feature` | _(none)_ |
| REQ-002 | _(none)_ | _(none)_ | _(none)_ |
```

**`traceability/test-map.md`** — the scenario-level chain, downstream links `pending`:

```
| Requirement | Test idea | BDD scenario | Automated test | Test result | Quality report |
| --- | --- | --- | --- | --- | --- |
| REQ-001 | CS-001-001 | `bdd/features/...feature` :: Expired session routes the holder back to sign-in | pending | pending | pending |
```

The `pending` cells are honest placeholders — Phase 6 fills Automated test, Phase 7 fills Test result and Quality report. The map never invents links. Full methodology: [`tc-traceability/methodology/traceability.md`](../../plugins/test-commander/skills/tc-traceability/methodology/traceability.md).

## What changed on disk

```
.test-commander/
  bdd/
    features/account-sign-in-with-email-and-password.feature   # generated
    summaries/account-sign-in-with-email-and-password.md        # generated + reviewed
    index.md                                                    # rebuilt
  traceability/
    requirements-map.md                                         # rebuilt (shared format)
    test-map.md                                                 # rebuilt (scenario chain)
  requirements/
    open-questions.md                                           # [bdd-review] signals (if any)
```

## Re-running

All three commands are idempotent and byte-deterministic: re-running against unchanged inputs reproduces identical files, and `[bdd-review]` gap signals are deduplicated. `/tc:traceability-map` is the authoritative regenerator of both maps from Phase 5 onward.

## Customizing for your project

Phase 5 ships universal tag classes and a universal review rubric. Projects pick `@area:`/`@risk:`/`@persona:` values and extend the rubric and class tags via `<workspace>/config.yaml` — see the "Phase 5 schema (`tc-bdd` / `tc-traceability`)" section of [customizing-for-your-project.md](customizing-for-your-project.md).

## Beyond Phase 5

Phase 6 builds the Playwright framework on demand and turns `@automated-candidate` scenarios into automated tests, filling the `Automated test` column of `test-map.md`. Phase 7 runs them and fills `Test result` and `Quality report`.
