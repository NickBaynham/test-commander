## [Unreleased]

### Phase 1 — Workspace and artifact model (in progress)

#### Added
- Step 1.1: workspace template — canonical `.test-commander/` workspace template that `/tc:init` copies into consuming projects in Step 1.2. 63 starter files: 35 directories, 33 explicitly-named files from the Workspace Layout (project.md, config.yaml, methodology.md, the 6 requirements files, the 9 product-knowledge files, the 3 traceability maps, the 4 learning files, policy/permissions.yaml + approvals.yaml, audit/actions.jsonl, etc.), plus per-directory README.md placeholders so every empty-by-design directory is reachable through git. Every Markdown starter has an H1 heading and a "Populated by `<command>` (Phase N)" note; YAML starters have the same shape in a `#` comment header; `audit/actions.jsonl` is empty by design (append-only log). Pre-flight `tests/test_workspace_template.py` (7 tests) covers structural completeness, Markdown heading + phase-note conventions, YAML comment-header conventions, and the empty-jsonl invariant.
- Step 1.2: `/tc:init` helper + command file. New Decision D18 — user-facing helpers and templates ship inside the plugin (`plugins/test-commander/scripts/` and `plugins/test-commander/templates/`); dev tooling stays at the repo root. Verified by inspecting the installed plugin cache. The template moved from `templates/workspace/` to `plugins/test-commander/templates/workspace/` so it travels with the plugin. `plugins/test-commander/scripts/init_workspace.py` is the helper: resolves the bundled template relative to its own location; copies into `<target>/.test-commander/`; idempotent (skips existing files, never overwrites); refuses targets that are regular files; supports a `--template` override. `plugins/test-commander/skills/tc-core/commands/init.md` is the per-command page (Inputs / Outputs / Preconditions / Behavior / Safety / Implementation / DoD / See also). Pre-flight `tests/test_init_workspace.py` (4 tests, test-first) covers fresh init, idempotent re-init, partial-existing, and refusal on invalid target. `pyproject.toml` pythonpath extended to include `plugins/test-commander/scripts`. The Phase 0 `test_no_command_behavior_yet` guard in `tests/test_plugin_scaffold.py` was removed (its purpose was to keep Phase 0 from accidentally implementing commands; Phase 1 now legitimately does). Smoke-tested end to end against a tmp dir: 63 created, 63 skipped on re-run, exit 2 on file target.
- Step 1.3: `/tc:status` helper + command file. `plugins/test-commander/scripts/workspace_state.py` produces a deterministic, typed `WorkspaceSnapshot` (workspace path, exists, initialized, last_modified, counts per bucket, populated counts per bucket, phase_status per phase). "Populated" = file bytes differ from the bundled template; orphan workspace files (no template counterpart) also count as populated. Phase status follows `PHASE_OWNERSHIP` — a phase is `in_progress` iff any owned file is populated, else `not_started`. Read-only — never writes. Shared with `/tc:next` in Step 1.5. CLI prints a grep-friendly report. `plugins/test-commander/skills/tc-core/commands/status.md` is the per-command page. Pre-flight `tests/test_workspace_state.py` (6 tests, test-first) covers empty workspace, fresh init, partial user content, full per-phase user content, snapshot determinism, and orphan-file detection. Smoke-tested end to end: 63 files / 0 populated after fresh init, 2 populated after touching project.md + requirements/requirements-inventory.md, phases 1 and 2 flip to in_progress while others stay not_started.
- Step 1.4: `/tc:journal` helper + command file. `plugins/test-commander/scripts/journal.py` ships two modes: `append` (timestamped entry to today's day file at `.test-commander/journal/YYYY-MM-DD.md`) and `summarize` (chronological entries within an inclusive `--from`/`--to` range; AI-generated summaries deferred to Phase 8). Day-file layout: H1 date header plus one H2 timestamp section per entry, written as valid Markdown. Append rejects empty bodies, bodies containing an H2 timestamp heading that would corrupt parsing, naive timestamps, and an uninitialized workspace. Summarize skips `README.md`, returns empty for a missing journal directory, and is read-only. CLI uses argparse subcommands (`append "body"` / `summarize --from YYYY-MM-DD --to YYYY-MM-DD`) with `--target` override. `plugins/test-commander/skills/tc-core/commands/journal.md` is the per-command page (Inputs / Outputs / Preconditions / Behavior / Safety / Out-of-scope / Implementation / DoD). Pre-flight `tests/test_journal.py` (8 tests, test-first) covers the 5 documented cases plus missing-dir summarize, malformed-body refusal, and uninitialized-workspace refusal. Smoke-tested end to end: two appends create a valid day file, summarize prints both in order, empty body exits 2 with a clear error.
- Step 1.5: `/tc:next` heuristics engine. New methodology doc `plugins/test-commander/skills/tc-core/methodology/next-step-inference.md` documents 10 R-rules (R1–R10) with trigger / recommendation / rationale / priority for each. The engine `plugins/test-commander/scripts/next_step.py` reads a `WorkspaceSnapshot` from Step 1.3's `workspace_state.snapshot()`, evaluates every rule, and returns matching `Recommendation(command, explanation, phase, priority)` records sorted by priority ascending. `format_recommendations()` renders the top match as `next: <command>  (Phase N)\n  <explanation>` and the rest under `followups:`. Read-only — never writes anything. Per-command page at `plugins/test-commander/skills/tc-core/commands/next.md` documents the rule table, output shape, and safety contract. Pre-flight `tests/test_next_step.py` (13 tests, test-first) provides one fixture per rule (R1–R10) plus three structural tests: ranked-list ordering, `next:` first line guarantee, and followups format. Smoke-tested end to end across three states: missing workspace → R1; fresh init → R2; after editing `project.md` → R3 with 6 followups. Phase 9 and Phase 10.5 deliberately have no R-rule yet (out-of-scope per methodology doc).
- Step 1.6: documentation pass. `docs/workspace-reference.md` filled in with the canonical `.test-commander/` layout, per-directory purpose, owning-phase notes, commit policy (per D5), and what-lives-outside-the-workspace section (test code at `tests/`, data via fixtures per D6). `docs/command-reference.md` rewritten as an index that links each Phase 1 command directly to its per-command page inside the plugin (`plugins/test-commander/skills/tc-core/commands/<name>.md`) plus a flat table of planned commands by phase. New `docs/user-guide/workflow.md` — end-to-end walkthrough of `/tc:init` → customize metadata → `/tc:status` → `/tc:journal` → `/tc:next` with realistic sample output for each. README status line updated from "Phase 1 starts next" to "Phase 1 in progress" and the documentation index gained the new workflow page. `docs/install.md` and `docs/user-guide/getting-started.md` "what's next" sections refreshed to reflect that Phase 1 commands are available. Link checker confirms all cross-links resolve (88 files, 0 broken).
- Step 1.7: testing finalization. `scripts/verify_skills.py` CATALOG entry for `tc-core` bumped from phase 0 to phase 1 and `DEFAULT_PHASE_CAP` bumped from 0 to 1, so `make verify` now prints `tc-core PRESENT (phase 1)`. New `tests/test_phase_1_integration.py` (1 test, 9 assertion blocks) drives every Phase 1 helper end-to-end against a fresh tmp consuming project: init (63 created) → re-init (idempotent: 0/63) → fresh-status snapshot (everything not_started) → next (R2 manual customize) → user edits project.md → status reflects in_progress for Phase 1 → next now recommends R3 (`/tc:review-requirements`) with downstream gaps as followups → two journal appends → summarize returns both chronologically → final status keeps Phase 1 in_progress with last_modified set. Complements the per-command unit tests by exercising composition; catches integration regressions that pass when each helper is tested in isolation.

### Phase 0 — Repository foundation (complete 2026-05-26)

Repository ready for Phase 1. End-to-end clean install verified: `./bootstrap.sh` → `make install` → `test-commander:tc-core` loaded in Claude Code. 46-test suite green; 23-file Markdown link check clean. Tagged `phase-0` on origin.

#### Added
- Step 0.1: MIT `LICENSE`, expanded `README.md`, `CONTRIBUTING.md`, this `CHANGELOG.md`, and `TODO.md` placeholder.
- Step 0.2: documentation skeleton under `docs/` (vision, architecture, roadmap, methodology, command-reference, workspace-reference, glossary, install) and `docs/user-guide/getting-started.md`. Markdown link checker at `scripts/check_links.py`.
- Step 0.3: Python project foundation. `pyproject.toml` (PDM-managed, `requires-python = ">=3.12"`, ruff and pytest dev deps, tool config inline). `Makefile` with `install`, `lint`, `test`, `build`, `run`, `verify` targets. `docker-compose.yml` placeholder. `.gitignore` for Python/PDM/IDE artifacts. `tests/test_placeholder.py` so pytest exits cleanly before real tests arrive.
- Step 0.4: POSIX `bootstrap.sh` that detects platform (macOS / Linux / WSL / Git Bash); verifies `git`, `make`, Python 3.12, PDM, Docker; auto-installs PDM via the official installer; auto-installs `git` and `make` on Linux/WSL via apt; prints suggested-install hints for the rest and exits non-zero. Idempotent. Never modifies `PATH` or writes a make shim. Bootstrap and `make install` are separate concerns — bootstrap verifies, install installs. Syntax-checked under both bash and dash.
- Step 0.5: plugin scaffold. `.claude-plugin/marketplace.json` declares `test-commander-marketplace` and references the in-repo `test-commander` plugin via `./plugins/test-commander`. `plugins/test-commander/.claude-plugin/plugin.json` declares version `0.0.0` with author, homepage, and repository fields. Plugin gets its own MIT `LICENSE` and a contents-focused `README.md`. `plugins/test-commander/skills/tc-core/SKILL.md` is the descriptor for the `tc-core` umbrella skill — describes `/tc:init`, `/tc:status`, `/tc:journal` and notes that command behavior arrives in Phase 1. Pre-flight test suite `tests/test_plugin_scaffold.py` covers all 8 automated DoD checks (file presence, JSON validity, manifest shape, frontmatter, command references, no-behavior-yet). Validated end-to-end with `claude plugin validate`, `claude plugin marketplace add`, and `claude plugin install`; `tc-core` appears as the sole skill in `claude plugin details test-commander`.
- Step 0.6: skill verifier. `scripts/verify_skills.py` walks `plugins/test-commander/skills/*/SKILL.md`, parses YAML frontmatter, and classifies each skill as PRESENT, MISSING, MALFORMED, or UNEXPECTED. `--phase N` filter restricts the expected set; default cap is 0 (bumped per phase as new skills land). Exit code 0 when every expected skill is PRESENT and nothing MALFORMED; non-zero otherwise; UNEXPECTED warns only. Includes a 20-entry catalog mirroring the TC-Owned Skill Catalog in the plan. Wired into `make verify` between `test` and `check_links`. Pre-flight test suite `tests/test_verify_skills.py` (16 tests) covers the parser, walker, phase filter, exit codes, and a live-run integration test. `pyproject.toml` gets `pythonpath = ["scripts"]` so pytest can import the verifier without sys.path hacks. All five live drills (baseline, --phase 0, --phase 2, MALFORMED corruption, MISSING rename) match expected output and exit codes.
- Step 0.7: `make install` wiring. Decomposed `install` into a five-step chain: `pdm-install` → `validate-manifests` → `marketplace-add` → `plugin-install` → `verify-skills`. Each step is idempotent: `validate-manifests` is read-only; `marketplace-add` and `plugin-install` guard against duplicate registration by checking the CLI lists first. New `make uninstall` target reverses the install and tolerates already-clean state via `-` prefix. `make help` updated. Pre-flight test suite `tests/test_make_install.py` (9 tests) covers target existence, dependency-chain order, the idempotency guards in each step, and `uninstall` tolerance. Documentation updates: `docs/install.md` now lists the five steps accurately and adds an uninstall subsection plus troubleshooting entries for "already registered/installed" and `validate-manifests` aborts; `docs/user-guide/getting-started.md` notes idempotency; plugin README aligned. Final DoD evaluation drill: `make uninstall` → `make install` from clean state (everything ✔), then `make install` again — no errors, exactly one marketplace entry, one plugin entry, `tc-core` PRESENT. Captured to `/tmp/tc-install-fresh.log` and `/tmp/tc-install-rerun.log`.
- Step 0.8: public-skill evaluation pass. `docs/skill-evaluation.md` (86 lines) evaluates the five candidate categories — Mermaid/diagram, devbox/sandbox (Coder, Daytona, Sprites.dev), traceability matrix, accessibility testing, performance testing — by scanning the two locally-installed Claude Code marketplaces (209 plugin entries combined). Each section captures What it does / Why interesting / Decision / Link. All five decisions land at "pass" — no dedicated plugin found in any category. Q1 (a11y/perf out of v1 scope), Q8 (docker-compose first sandbox), and Q12 (build `tc-visualize` ourselves) defaults stand. `chrome-devtools-mcp` noted as a future browser-perf reference. Pre-flight `tests/test_skill_evaluation.py` (4 tests) asserts file existence, line cap, all 5 categories present, and all 4 required fields per section. Cross-linked from `docs/methodology.md` and the README documentation index.
- Step 0.9: Phase 0 sign-off. Cold-user smoke test of `docs/user-guide/getting-started.md` from a clean state (`make uninstall` → `bootstrap.sh` → `make install`) succeeded end to end; output captured to `/tmp/tc-phase0-walkthrough.log`. Per-step DoD audit confirmed every Step 0.1–0.8 deliverable on disk. `planning/plan.md` To Do Phase 0 collapsed to a "see Completed" marker; Completed section populated with the nine sub-step summaries and the 2026-05-26 date. CHANGELOG Phase 0 heading marked complete. `tests/test_phase_0_signoff.py` (5 tests) asserts all five Phase 0 pytest files exist, CHANGELOG and plan are properly marked, and the suite has at least 41 tests. Annotated tag `phase-0` created and pushed to origin.
